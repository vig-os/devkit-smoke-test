In the container-based `rollback` job, `git` commands are executed after checkout, but there is no Git safe.directory configuration. Given other container workflows here explicitly run `git config --global --add safe.directory "$GITHUB_WORKSPACE"`, add the same after checkout to avoid Git refusing to operate due to “dubious ownership” in container mounts.
```suggestion
        run: |
          git config --global --add safe.directory "$GITHUB_WORKSPACE"
```

`release.yml` now delegates most work to reusable workflows, but the caller jobs (`core` / `publish`) don't declare the permissions those called workflows need. For reusable workflows, the called workflow’s `GITHUB_TOKEN` permissions are capped by the *caller job’s* `permissions`; with workflow-level `contents: read` this will make release-core unable to write finalization commits or read PR metadata, and it will prevent tag/release publishing. Declare required permissions on the `core` job (at least `contents: write`, plus `pull-requests: read` and `actions: write` if using `gh pr ...` / `gh workflow run`), or pass a PAT/App token via the reusable workflow `secrets.token` input and stop relying on the default token.

```diff
+    container:
+      image: ghcr.io/vig-os/devcontainer:${{ needs.resolve-image.outputs.image-tag }}
+    timeout-minutes: 10
+    permissions:
+      contents: write
+    outputs:
+      tag: ${{ steps.out.outputs.tag }}
+      release_url: ${{ steps.out.outputs.release_url }}
+
+    steps:
+      - name: Resolve auth token
+        id: auth
+        env:
+          PROVIDED_TOKEN: ${{ secrets.token }}
+          FALLBACK_TOKEN: ${{ github.token }}
+        run: |
+          set -euo pipefail
+          if [ -n "${PROVIDED_TOKEN:-}" ]; then
+            echo "token=$PROVIDED_TOKEN" >> "$GITHUB_OUTPUT"
+          else
+            echo "token=$FALLBACK_TOKEN" >> "$GITHUB_OUTPUT"
+          fi
+
+      - name: Validate release kind
+        env:
+          RELEASE_KIND: ${{ inputs.release_kind }}
+        run: |
+          set -euo pipefail
+          if [ "$RELEASE_KIND" != "candidate" ] && [ "$RELEASE_KIND" != "final" ]; then
+            echo "ERROR: Invalid release_kind '$RELEASE_KIND'"
+            echo "release_kind must be one of: candidate, final"
+            exit 1
+          fi
+
+      - name: Checkout finalized commit
+        uses: actions/checkout@de0fac2e4500dabe0009e67214ff5f5447ce83dd  # v6.0.2
+        with:
+          ref: ${{ inputs.finalize_sha }}
+          fetch-depth: 0
+          token: ${{ steps.auth.outputs.token }}
+
```

`publish` runs in a container and performs several `git` operations (tagging, pushing). Other container-based jobs in this repo add `git config --global --add safe.directory "$GITHUB_WORKSPACE"` after checkout to avoid Git’s “dubious ownership” failures, but this workflow doesn’t. Add that safe.directory configuration right after `Checkout finalized commit` to prevent container-only git errors.
```suggestion

      - name: Mark workspace as safe directory
        run: |
          git config --global --add safe.directory "$GITHUB_WORKSPACE"
```
